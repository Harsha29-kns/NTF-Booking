Security Implementation Report

Target Threat: Ticket Duplication SOLVED

The Threat: Attackers attempting to gain entry using the same ticket multiple times (Replay Attack) or by sharing screenshots of a valid QR code with others (Cloning).
Executive Summary: The system now employs a Hybrid Defense Mechanism combining Dynamic QR Codes (Time-based One-Time Passwords) for the frontend and Real-Time Centralized Logging for the backend. This effectively neutralizes both offline screenshot sharing and parallel gate entry attempts.

Implemented Solutions

1. Dynamic QR Codes (Anti-Screenshot)

Static QR codes are vulnerable to being screenshot and shared. We have implemented Time-Based Dynamic QRs:

  • Mechanism: The User's "Print Ticket" page auto-regenerates the QR code payload every 15 seconds.
  • Payload: Contains { ticketId, buyerAddress, timestamp, signature }.
  • Defense: If a user screenshots the code and sends it to a friend, by the time the friend reaches the gate, the 15-second window has passed, and the scanner will reject the timestamp as EXPIRED.

2. Real-Time Entry Logging (Anti-Replay)

To prevent the same valid ticket from being used twice (e.g., at two different gates simultaneously):

  • Mechanism: Upon a successful scan, the Backend immediately records an entry log in the secure MongoDB database.
  • Instant Check: Before granting access, the system checks: Has this ticketId been scanned in the entry logs?
  • Result: If a duplicate is found, the Gatekeeper is instantly alerted with a DUPLICATE SCAN or ALREADY USED error, including the timestamp of the previous entry.

3. Hybrid Verification Architecture

Process Flow:
[QR Scan] ➔ [Check Blockchain Ownership] ➔ [Check DB for Prior Entry] ➔ [Grant Access]

We utilize the Blockchain for what it does best (Ownership Security) and the Database for what it does best (Speed & Anti-Replay):

  • Blockchain (Solidity): Verifies that the ticket is authentic and owned by the user. (Immutable Source of Truth)
  • Database (MongoDB): key functionality for "Mark as Used". This avoids the 15-second block confirmation delay and gas fees, complying with venue requirements for rapid throughput.
